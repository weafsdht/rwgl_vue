# 前端需完善内容

本文档列出 TaskFlow 前端（Vue 3 + TypeScript + Pinia + Naive UI）为对接当前后端**需要完善或新增**的内容，便于按项实现或联调。

---

## 一、必修（Bug 与联调）

### 1. 归档页：`selectedIds.has is not a function`

- **位置**：`Archive.vue`（或归档相关组件）
- **问题**：`selectedIds` 被当作 `Set` 使用（`.has()`），实际可能是数组。
- **处理**：二选一  
  - 用 `ref<Set<number>>(new Set())`，勾选/取消用 `add/delete`，判断用 `.has(id)`；  
  - 或继续用数组，判断改为 `selectedIds.includes(id)`，不要调用 `.has()`。
- **详见**：`frontend/FRONTEND_FIXES.md` 第 1 节。

### 2. 刷新看板数据消失

- **问题**：请求失败时把看板清空，导致刷新后四列变 0。
- **处理**：调用 `fetchBoard` 时传入 **`getCurrentBoard`**，例如 `getCurrentBoard: () => board.value`（board 为当前看板状态）。这样仅在「首次加载且无数据」时在失败时设为空，刷新失败时保留当前数据。
- **详见**：`frontend/FRONTEND_FIXES.md` 第 2.1 节；本仓库 `frontend/src/stores/task.ts` 已支持该参数。

### 3. 网络/鉴权与 baseURL

- **现象**：Network Error、获取用户信息失败、上传失败等。
- **处理**：  
  - 确认 baseURL 与后端一致（本机 `http://localhost:8080`；局域网其他设备用 `http://192.168.1.220:8080` 等）。  
  - 先访问 `http://<后端地址>/api/health` 验证可达；请求头带 `Authorization: Bearer <token>`。  
  - 错误提示建议展示 `response.data.message`，而不是统一“服务器内部错误”。
- **详见**：`frontend/FRONTEND_FIXES.md` 第 2、3、5 节。

### 4. 附件上传请求格式

- **接口**：`POST /api/tasks/{taskId}/attachments`
- **要求**：请求体 **multipart/form-data**，字段名必须为 **`file`**。
- **示例**：`formData.append('file', file)`；不要用 `attachment`、`files` 等其它字段名。
- **预览/下载**：后端返回的 `fileUrl` 可能为相对路径，预览与下载需用 `getAttachmentUrl(fileUrl)`（与 `getAvatarUrl` 同逻辑，拼 `getBackendOrigin()`）；前后端不同域时配置 `VITE_API_BASE` 或 `VITE_API_BASE_URL`，详见 `frontend/FRONTEND_FIXES.md` 第 5.1 节。
- **详见**：`frontend/FRONTEND_FIXES.md` 第 5 节。

### 5. 附件下载（避免文件损坏）

- **问题**：未检查 HTTP 状态直接 `response.blob()`，4xx/5xx 时可能把错误页 HTML 当文件保存。
- **处理**：`fetch` 或 `axios` 拿到响应后先判断 **`response.ok`**（或 axios 的 `status` 在 200–299），再 **`await response.blob()`** 并触发下载；失败时提示后端错误信息，不要写入本地文件。

### 6. 任务负责人 `assigneeIds`

- **类型**：传 **用户数字 id 数组**（如 `GET /api/users/me` 返回的 `data.id`、用户列表中的 `id`）。
- **禁止**：把 **邮箱字符串** 当作 `assigneeIds` 元素传入（除非后端明确支持，当前约定为 id）。

### 5. 用户管理与角色（仅系统管理员）

后端已提供角色：0=系统管理员、1=项目经理、2=普通用户。GET /api/users/me 及登录/注册返回的 User 中带 **role**、**roleLabel**。以下为前端实现要点。

**一、当前用户（鉴权与展示）**

- **类型**：在 `src/types/index.ts` 的 User 上增加可选字段：`role?: number`（0/1/2）、`roleLabel?: string`（展示用）。
- **数据来源**：由 GET /api/users/me 以及登录/注册返回的 User 带入；后端已在 UserVO 中返回 role、roleLabel。

**二、用户列表与管理员操作接口**

- **类型**：在 `src/types/index.ts` 新增 **AdminUserVO**：`id`、`uid?`、`email`、`username`、`nickname?`、`role`、`roleLabel`（或直接使用本仓库 `frontend/src/api/admin.ts` 中的 `AdminUserVO` 类型）。
- **API**：`src/api/admin.ts`（本仓库已提供：`frontend/src/api/admin.ts`）
  - `getUsers(api)` → GET /api/admin/users，返回 `AdminUserVO[]`
  - `updateRole(api, id, role)` → PATCH /api/admin/users/{id}/role，body 为 `{ role: number }`
  - **`updatePassword(api, id, password)`** → PATCH /api/admin/users/{id}/password，body 为 `{ password: string }`（至少 6 位）
  - **`deleteUser(api, id)`** → DELETE /api/admin/users/{id}
- 上述函数第一个参数 `api` 为你的 HTTP 客户端（需包含 `get`、`patch`、`delete` 方法，且已带鉴权与 baseURL）。

**三、角色选项与权限判断**

- **常量**：`src/constants/role.ts`（本仓库已提供：`frontend/src/constants/role.ts`）
  - `ROLE_OPTIONS`：`[{ value: 0, label: '系统管理员' }, ...]`
  - **`isSystemAdmin(role)`**：支持 **`number | string | undefined`**，内部用 **`Number(role) === 0`** 判断。这样后端返回 `role: 0` 或 `role: "0"` 都会被视为系统管理员。
  - **`isProjectManagerOrAdmin(role)`**：**`Number(role) === 0 || Number(role) === 1`**，用于控制「项目管理、统计模块、任务关联项目」的菜单与操作（仅项目经理或系统管理员可见/可用）。
  - `getRoleLabel(role)`：按 role 返回对应文案（同样兼容 number/string）。

**四、权限控制（路由与菜单统一用同一套判断）**

- **路由**：新增路由 `path: 'admin/users'`，`name: UserManagement`，`meta: { requiresAdmin: true }`。
- **守卫**：在 `router/index.ts` 的 `router.beforeEach` 中，若 `to.meta.requiresAdmin` 且 **`!isSystemAdmin(userStore.user?.role)`**，则重定向到 `/`。只在一处导入 `isSystemAdmin`，避免重复。
- **菜单**：侧栏「用户管理」的显示条件与路由一致，使用 **`isSystemAdmin(userStore.user?.role)`**，这样与守卫逻辑统一。

**若看不到「用户管理」请排查：**

1. **刷新页面或重新登录一次**，让前端用最新的 user（含 role）再算一遍菜单和路由。
2. 确认当前登录账号在后端用户表里 **role 为 0**（或字符串 `"0"`）。
3. 确认登录或 **GET /api/users/me** 的返回里 **包含 `role` 字段**。
4. 可在控制台执行下面代码，确认接口是否带了 role（替换为你的 userStore 变量名）：
   ```js
   // 在控制台执行（若 user 在 window 上暴露了可改为 window.user）
   const user = useUserStore().user;
   console.log('user.role', user?.role, 'type', typeof user?.role, 'isSystemAdmin', isSystemAdmin(user?.role));
   ```

**五、用户管理页（列表 + 修改角色 + 修改密码 + 删除用户）**

- **页面**：`src/views/UserManagement.vue`
- 进入时请求 GET /api/admin/users 展示表格。
- **表格列**：邮箱、用户名、昵称、角色（roleLabel）、操作。
- **操作列**：每行三个按钮——**「修改角色」**、**「修改密码」**、**「删除」**。
  1. **修改角色**：点击后弹窗内用 ROLE_OPTIONS 下拉选择角色，确认后调用 `updateRole(api, id, role)`，成功后刷新列表并提示成功。
  2. **修改密码**：点击后打开弹窗，展示对应用户信息（如邮箱/用户名），输入新密码（至少 6 位），确定后调用 `updatePassword(api, id, password)`，成功则刷新列表并提示成功。
  3. **删除用户**：点击后打开确认弹窗，文案为「确定要删除用户「xxx」吗？此操作不可恢复。」（xxx 可为该行用户名或邮箱），确认后调用 `deleteUser(api, id)`，成功则刷新列表并提示成功。

**后端约定（已实现）**：GET /api/users/me 返回的 User 含 role、roleLabel；GET /api/admin/users 仅系统管理员可调，返回 AdminUserVO[]；PATCH /api/admin/users/{id}/role 的 body 为 `{ role: number }`（0/1/2）；**PATCH /api/admin/users/{id}/password** 的 body 为 `{ "password": "新密码" }`（至少 6 位）；**DELETE /api/admin/users/{id}** 删除指定用户（仅系统管理员，且后端禁止删除当前登录用户）。

## 三、接口约定（对接时统一）

- **基础路径**：`/api`；若 axios baseURL 为 `http://localhost:8080`，则路径为 `/api/tasks` 等。
- **任务列表**：`GET /api/tasks` 返回的每条任务已带 **projectId**、**projectName**、**projectColor**（后端按 projectId 自动填充），前端可在表格中增加「项目」列，展示 `projectName`，并用 `projectColor` 做色标。
- **看板**：`GET /api/tasks/board` 各列中的任务同样包含 **projectName**、**projectColor**，卡片上可展示所属项目。
- **鉴权**：请求头 `Authorization: Bearer <token>`。
- **响应**：`{ code: 0, message: "success", data: ... }`，业务数据在 `data`；列表/看板等均为 `data` 内结构。
- **当前用户**：`GET /api/users/me` 与登录/更新资料返回的 **UserVO** 包含：`id`、`email`、`username`、`nickname`、`avatar`、`phone`、`timezone`、**`role`**、**`roleLabel`**。设置页可展示 `id`、`email`，编辑 `username`、`nickname`、`phone`、`timezone`，头像通过 `POST /api/users/me/avatar` 上传。`role` 用于权限控制（如仅系统管理员可见用户管理），`roleLabel` 用于展示。
- **项目经理与系统管理员（项目管理、统计、任务分配）**：仅 **系统管理员（role=0）** 或 **项目经理（role=1）** 可：**管理项目**（创建/更新/删除项目、里程碑、共享与编辑申请、项目进度/统计）、**使用统计模块**（GET /api/statistics 及 /completion-rate、/dashboard 等子路径）、**将任务关联到项目**（创建或更新任务时设置 projectId）。普通用户（role=2）调用上述接口将收到 **403**。前端应根据 `role` 隐藏或禁用：创建项目按钮、项目编辑/删除、里程碑与共享管理入口、统计菜单或统计页、任务表单中的「所属项目」选择（普通用户不显示项目选择或仅展示不可编辑）。
- **任务状态**：`status` 为 0-based：0=待办、1=进行中、2=完成、3=搁置；拖拽或修改状态时传 0–3。
- **项目颜色**：项目接口返回的 `color` 与 `cover` 同值，前端可用 `data.color ?? data.cover` 渲染项目色条/主题色。

- 提供「新建标签」按钮，点击后弹窗或内联表单输入名称、选择颜色，提交调用 POST /api/tags，成功后刷新列表。

### 7. 项目详情与编辑回填

**用途**：后端已提供 `GET /api/projects/{id}`，返回 `assigneeIds`、`assignees`、`teamIds`、`teams`、`myPermission` 等。列表接口若不带负责人/团队，编辑弹窗可通过详情接口在打开后异步回填，保证负责人与关联团队多选正确显示并提交。

**类型（与后端 ProjectVO 对齐）**：`ProjectVO`、`ProjectAssigneeVO`、`ProjectTeamBasicVO`，见 `src/types/index.ts`；含 `assigneeIds`、`assignees`、`teamIds`、`teams`、`myPermission` 等。

**API**：`src/api/project.ts`
- **getProject(api, id)**（命名导出）：入参 `api` 为仅需具备 `get(url)` 的客户端（如 axios 实例或带鉴权的 request 封装）。请求 `GET /api/projects/{id}`，从 `response.data` 解包（支持 `{ code, message, data }` 或直接为业务对象）后返回 `ProjectVO`，供编辑回填使用。
- 本仓库内部使用默认导出的 `projectApi.getProject(id)`（已基于 request 解包），无需传 api；实际接入时若使用自定义客户端，可将示例中的 `api` 换成你的 axios 实例。

**openEdit 四步流程**（`src/views/Projects.vue`）：
1. **同步**：用当前列表项 `project` 立刻给表单赋值（name、description、color、assigneeIds、teamIds）并打开弹窗，设置 `editingProjectId = project.id`。
2. **异步**：在后台调用 `projectApi.getProject(project.id)`（或 `getProject(api, project.id)`）。
3. **仍编辑同一项目时**：若 `editingProjectId === detail.id`，用详情的 `assigneeIds`、`teamIds` 覆盖表单（可选再同步 `assignees`、`teams` 做多选回显）。
4. **详情失败**：`getProject` 抛错（如 404 或接口不存在）时在 try/catch 中静默处理，不提示错误，表单继续使用步骤 1 的列表数据。

**失败不报错策略**：避免因暂无详情接口或 404 打断用户编辑，列表已有的 name/description/color 仍可修改并提交。

本仓库中 `src/api/project.ts` 与 `src/views/Projects.vue` 为可复用示例；后端已提供 `GET /api/projects/{id}` 并返回 `assigneeIds`、`assignees`、`teamIds`、`teams` 时，前端接入上述 API 与 openEdit 逻辑即可在列表未带负责人/团队时也在编辑弹窗中正确回填并提交。

### 8. 消息通知模块

**功能**：及时的信息推送与提醒（任务变化、项目进度等）；多种通知类型（任务分配、状态变更、截止提醒、系统公告等）；按紧急程度与类型分类展示；已读/未读与历史记录；用户可选站内消息、邮件（后端支持偏好，邮件发送可后续对接）；智能频率控制（同类型防刷间隔可配置）。

**接口**：

| 接口 | 说明 |
|------|------|
| GET /api/notifications | 列表，可选参数 isRead、type、urgency、limit；返回项含 typeLabel、urgencyLabel |
| GET /api/notifications/unread-count | 未读数量，data.count |
| POST /api/notifications/mark-all-read | 全部标记已读 |
| PATCH /api/notifications/{id}/read | 单条标记已读 |
| GET /api/notifications/preferences | 当前用户偏好：channelInApp、channelEmail、frequencyMinutes |
| PATCH /api/notifications/preferences | 更新偏好（body: channelInApp、channelEmail、frequencyMinutes） |

**类型 type**：1=截止提醒 2=提前提醒 3=每日计划 4=未完成提醒 5=任务分配 6=状态变更 7=系统公告。**紧急程度 urgency**：0=普通 1=重要 2=紧急。前端可按类型、紧急程度筛选与分组展示；未读数用于角标；设置页可提供「通知偏好」入口。

### 9. 任务与项目：负责人/团队 → 共享 + 通知

**后端已实现**：任务和项目在选择了负责人或团队时，会自动完成共享与站内通知。

**项目**
- 创建/更新项目时传 **assigneeIds**（负责人用户 ID 列表）、**teamIds**（关联团队 ID 列表）。
- 后端会：将负责人写入 project_assignee 并同步到 project_share（编辑权限）；对每位负责人（排除当前操作人）发送站内通知「xxx 将您添加为项目《项目名》的负责人」；为团队内每位成员写入 project_share（只读）并发送「项目《项目名》已共享给您的团队」（排除当前操作人；同一用户若既是负责人又在团队内，只按负责人通知一条）。

**任务**
- 创建/更新任务时传 **assigneeIds**（负责人用户 ID 列表）。POST /api/tasks、PATCH /api/tasks/{id} 的 body 支持 assigneeIds。
- 后端会：写入 task_assignee；对每位负责人（排除当前操作人）发送站内通知「xxx 将任务《任务名》分配给您」（type=任务分配）；任务列表、看板、归档、按时间范围/按项目统计、详情与更新、恢复归档均按「创建人或任一负责人」可见/可操作。

**前端现状（本仓库）**
- **项目**：`src/views/Projects.vue` 新建/编辑表单已包含负责人多选（assigneeIds）、关联团队多选（teamIds），提交时一并传给后端，无需改动。
- **任务**：`src/components/CreateTaskModal.vue` 表单含负责人多选（assigneeIds），创建与更新时 `buildPayload()` 已传 `assigneeIds`（及兼容的 assigneeId），无需改动。
- **列表/看板/归档**：后端已按「创建人或负责人」过滤返回，前端无需改列表请求参数。

## 八、好友与私聊（聊天）

> **说明**：钉钉式私聊；后端已就绪，**原文档此前未单独成节**，以下便于前端对照实现。

### 1. 前置：好友

- 仅 **好友** 可发起私聊；需先具备 **好友列表 / 加好友**（`GET /api/friends`、`POST /api/friends/requests/by-email` 等，见下文接口表）。
- **打开会话**：使用 **`peerUserId`**（用户表主键，数字 id）；若界面仅有展示字段，**`peerEmail`** 更稳（与后端 `POST /api/chat/conversations/open` 约定一致）。

### 2. 前端待做（产品）

| 项 | 说明 |
|----|------|
| **路由与入口** | 侧栏或「消息」入口 → **会话列表** → **单聊页** |
| **会话列表** | 展示最后一条预览、时间、**未读数**、**对方是否已读我最后一条**（依接口字段） |
| **消息区** | 文本气泡；图片/文件气泡；图片可预览；文件可下载（**`fileUrl` 拼后端域名**，与任务附件 `getAttachmentUrl` / `getBackendOrigin` 一致） |
| **已读** | 进入会话或滚到底调用 **标记已读**；单条旁「已读」展示可依据 **`readByPeer`** 等返回字段 |
| **实时** | **SockJS + STOMP** 连接 **`/ws?token=<JWT>`**，订阅 **`/user/queue/chat`**，处理 **`NEW_MESSAGE`**、**`READ_RECEIPT`** 等帧 |

### 3. 聊天 HTTP 接口（路径均相对带 `/api` 前缀的 baseURL）

| 方法 | 路径 | 说明 |
